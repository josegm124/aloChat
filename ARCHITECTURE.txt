2. What the MVP does

The MVP evaluates AI-system documentation for regulated use cases, starting with Healthcare and HR, and produces:
- a structured assessment
- a document analysis with compliance findings
- a final web report
- a downloadable PDF report
- a notification payload

Cloud mode uses AWS-managed services.
Local mode replaces the paid AWS dependencies with local containers so daily development does not require AWS spend.

3. Backend services

- `alo-intake-service` (`:8080`)
  - accepts `POST /api/v1/assessments`
  - persists assessment intake and publishes the intake event
- `alo-profile-service` (`:8081`)
  - resolves regulatory profile and applicability
- `alo-document-compliance-service` (`:8082`)
  - accepts `POST /api/v1/document-compliance/analyze`
  - extracts PDF text and generates findings/evidence
- `alo-dataset-compliance-service` (`:8083`)
  - dataset-focused analysis
- `alo-assessment-orchestrator` (`:8085`)
  - consolidates results and requests report generation
- `alo-report-service` (`:8086`)
  - generates the final HTML/PDF report
  - exposes:
    - `POST /api/v1/reports/generate`
    - `POST /api/v1/reports/finalize`
    - `GET /api/v1/reports/{reportId}`
    - `GET /api/v1/reports/{reportId}/view`
    - `GET /api/v1/reports/{reportId}/pdf`
- `alo-notification-service` (`:8087`)
  - accepts `POST /api/v1/notifications/generate`
  - sends via SES in cloud mode
  - returns `LOCAL_DISABLED` in local mode
- `alo-retry-router-service` (`:8088`)
  - reserved for deferred retry routing

4. Event topology

Kafka topics in the current design:
- `alo.assessment.intake.received`
- `alo.assessment.profile.resolved`
- `alo.assessment.document.requested`
- `alo.assessment.dataset.requested`
- `alo.assessment.document.completed`
- `alo.assessment.dataset.completed`
- `alo.assessment.report.requested`
- `alo.assessment.notification.requested`
- `alo.assessment.notification.completed`
- `alo.retry.short`
- `alo.retry.long`
- `alo.dlq`

Current resilience strategy:
- immediate consumer retries
- poison-message handling with `ErrorHandlingDeserializer`
- DLQ routing to `alo.dlq`
- Kafka publisher circuit breaker

6. Local architecture

Local MVP mode replaces the expensive AWS pieces with:
- Redpanda instead of MSK
- DynamoDB Local instead of DynamoDB
- MinIO instead of S3
- local-disabled notification instead of SES

Local Docker services:
- `redpanda`
- `redpanda-init`
- `dynamodb-local`
- `dynamodb-init`
- `minio`
- `minio-init`
- `alo-intake-service`
- `alo-document-compliance-service`
- `alo-report-service`
- `alo-notification-service`

Local ports:
- frontend: `127.0.0.1:3000`
- intake: `localhost:8080`
- document: `localhost:8082`
- report: `localhost:8086`
- notification: `localhost:8087`
- DynamoDB Local: `localhost:8000`
- MinIO API: `localhost:9000`
- MinIO console: `localhost:9001`
- Redpanda external listener: `localhost:19092`

Local buckets created automatically:
- `alo-uploads`
- `alo-reports`

9. CI/CD

Current GitHub Actions structure:
- `CI`
  - runs on PRs and pushes to `master`
  - runs tests/build only
- `CD Dev`
  - runs after CI succeeds
  - intended for the AWS dev runtime
  - uses GitHub OIDC and the AWS role configured for the repo

Implication:
- pushing to `master` can trigger deployment work
- local MVP changes should stay on `feature/RBSai` until deliberately merged
